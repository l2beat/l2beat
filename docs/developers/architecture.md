# Architecture

DeFiScan V2 builds on the existing architecture and extends it with tooling dedicated to DeFi monitoring and full inventory tracking of protocol assets.

![DeFiScan V2 Architecture](../assets/DeFiDisco-architecture.svg)

DeFiScan V2 adds:

- Detection and analysis of function-level permissions (`functions.json`)
- A call graph of external contract calls, built with Slither (`call-graph-data.json`)
- Tracking of funds and DeFi positions controlled by the protocol (`funds-data.json`)
- Researcher-authored contract tags (`contract-tags.json`)

## Frontend

The researcher UI (`protocolbeat`) is a React application that visualizes discovery output and drives the manual review process. It queries a local Express API defined in `packages/l2b/src/implementations/discovery-ui/main.ts`. Following the [minimal integration principle](contributing.md), all DeFiScan V2 endpoints live in `packages/l2b/src/implementations/discovery-ui/defidisco/` and are imported into `main.ts`; frontend API calls live in `packages/protocolbeat/src/api/api.ts`.

`main.ts` calls `dotenv.config()` at module entry, so `l2b ui` started from `packages/config/` automatically picks up the project `.env` (used by `DATABASE_URL`, AI provider keys, etc.) without requiring shell exports.

## Backend

The backend exposes API endpoints that both return analysis data and run the analysis routines themselves (discovery, funds fetching, call graph generation, AI permission detection, review compilation). It also handles researcher actions such as tagging contracts and overriding function characteristics.

### Discovery

The discovery process is mostly untouched from upstream L2BEAT. It uses `config.jsonc` and contract templates to determine how each contract is scanned, and writes results to `discovered.json`. This output is the foundation for every other analysis stage.

### Funds Data

Funds data is fetched through a separate [defiscan-endpoints](https://github.com/deficollective/defiscan-v2/tree/main/packages/defiscan-endpoints) service. The backend calls this service and stores the result in a `funds-data.json` file per project. DeFiScan V2 supports token balances and DeFi positions for arbitrary contracts, token market cap for tokens, and aggregate TVL for factory contracts (via The Graph subgraphs and protocol-specific APIs).

See [Infrastructure: Funds Tracking](features/infrastructure.md#funds-tracking) for details.

### Call Graph

The call graph tool deterministically detects every external contract call made by each function of a contract and — where possible — maps those calls back to known contracts in the discovery. Resolution typically succeeds for 10–30% of calls; the rest are non-deterministic and deliberately left unresolved. Detection is built on Slither's SlithIR output, parsed in `callGraph.ts` and stored in `call-graph-data.json`.

See [Call Graph Analysis](features/call-graph-analysis.md).

### Permissions

Permissions are detected via AI queries. Each query sends the contract source code with a prompt asking for permissioned functions; results are filtered against the contract's ABI to remove hallucinations. Alongside each function, the model resolves the **owner** — a path expression into `discovered.json`:

- `$self` — the current contract itself
- `$self.fieldName` — a field of the current contract (e.g. `$self.owner`)
- `$self.signers[0]` / `$self.signers` — an array entry or the full array
- `$self.accessControl.ADMIN_ROLE.members` — members of an AccessControl role
- `@governor.fieldName` — follows a field on the current contract to another contract, then navigates further

Results are stored in `functions.json` per project, grouped by contract, and can be overridden by researchers in the UI. See [Permissions](features/permissions.md).

### Contract Tags

Contract tags are researcher-authored: they mark a contract as external to the project, tag it as governance, assign it to a provider (`Chainlink`, `Uniswap`, …), and toggle whether funds or positions should be fetched for it. Stored in `contract-tags.json` per project. See [Infrastructure: External Contract Attributes](features/infrastructure.md#external-contract-attributes).

### Review Builder

The Review Builder stores review content across **three sibling files** per project, intentionally split so AI-regenerated content cannot clobber hand-authored data:

- **`review-config.json`** — Protocol metadata, entity descriptions for admins/dependencies/fund-holding contracts, and section content (e.g. Code & Audits). Wiped and regenerated by the review-generation workflow.
- **`resources.json`** — Resources (frontends, docs, GitHub, licenses, …), security audits, and the deduplicated lines-of-code count. Shape: `{ resources, audits, linesOfCode? }`. Survives regeneration.
- **`governance.json`** — Governance configuration (framework, voting, proposal requirements, durations). Survives regeneration.

Each file has its own CRUD module under `packages/l2b/src/implementations/discovery-ui/defidisco/` (`reviewConfig.ts`, `resources.ts`, `governance.ts`) and its own HTTP endpoints. Legacy configs that still carry governance or resources inline in `review-config.json` are read transparently and migrated to the split layout on first write.

See [Scoring & Review: Review Builder](features/scoring-and-review.md#review-builder).

### Continuous Monitoring Service

The monitoring service is a standalone background process that watches each listed DeFi project for contract changes, refreshes live financial data, and produces publishable review artifacts. It runs as a **GitHub Actions cron job** (daily), not as a long-running server.

**Entry point:** `packages/backend/src/defidisco-monitor.ts`
**Orchestrator:** `packages/backend/src/modules/defi-update-monitor/defidisco/DefidiscoMonitorApplication.ts`

For each project in `packages/config/src/defidisco-config.json`, a monitor cycle:

1. Runs L2BEAT discovery
2. Diffs against the previous snapshot stored in PostgreSQL
3. Sends a Discord notification if contract changes are detected
4. Writes the fresh `discovered.json` back to the project folder so downstream consumers stay in sync
5. Refreshes funds data (balances, DeFi positions, aggregate TVL)
6. Reconciles the per-project `activity.json` feed from new `UpdateNotifier` rows
7. Recompiles `compiled-review.json`

The compilation step is gated on `review-config.json` and `call-graph-data.json` being present; if either is missing, compilation is skipped silently.

See [Infrastructure: Continuous Monitoring Service](features/infrastructure.md#continuous-monitoring-service).

### Monitor Admin Dashboard

The Monitor Admin Dashboard (`/ui/monitor-admin`) is a researcher cleanup UI for the `UpdateNotifier` Postgres table populated by the monitoring service. It exists because monitor cycles inevitably catch noisy ticking values that snuck past `ignoreInWatchMode`, and the only previous remedy was hand-editing the database and `activity.json`.

The page lists every persisted monitor cycle row grouped by project. Each row can be **fully deleted** or have **individual fields stripped**. Both actions cascade into `activity.json` (keeping the live feed consistent) and recompile the project's review. An optional toggle also promotes stripped fields into `overrides[address].ignoreInWatchMode` in `config.jsonc` so the same field never alerts again.

All endpoints are gated on `process.env.DATABASE_URL`. When unset, the page shows an "unavailable" notice and the endpoints return `503`. The `l2b ui --readonly` flag makes mutations return `403`.

See [Infrastructure: Monitor Admin Dashboard](features/infrastructure.md#monitor-admin-dashboard).

### DeFiScan Frontend

The public-facing review website is a separate package at `packages/defiscan-frontend/`. It is a static React application (Vite + Tailwind + Recharts) that renders compiled reviews. It never talks to the l2b API at runtime.

1. `ReviewCompiler` produces `compiled-review.json` per project
2. Compiled reviews are placed under `packages/defiscan-frontend/public/data/<slug>/`
3. A build-time script (`scripts/compile-data.ts`) aggregates them into `public/data/index.json` with global stats and cross-protocol dependency data

Pages: **Landing** (protocol table and global stats), **Gallery** (card grid with radar charts and filters), **Review** (Report / Explorer / Activity views), **Compare** (side-by-side protocol comparison), **About**. See `packages/defiscan-frontend/README.md` and [Infrastructure: DeFiScan Frontend](features/infrastructure.md#defiscan-frontend).

## Data Pipeline

The full pipeline, from source data files to the rendered frontend, is:

```
SOURCE DATA (per project)
├─ discovered.json            L2BEAT discovery output
├─ functions.json             Permissioned functions + owner paths + scores
├─ call-graph-data.json       Slither-based external call graph
├─ funds-data.json            Balances, positions, token market cap
├─ contract-tags.json         External/governance flags + entity
├─ review-config.json         Metadata + entity descriptions
├─ resources.json             Resources + audits + lines of code
├─ governance.json            Governance configuration
└─ activity.json              Monitor-reconciled activity feed
         │
         ▼
STAGE 1 — ProjectAnalysis (projectAnalysis.ts)
  getAdmins(), getDependencies(), getSummary()
  Orchestrates: enhancedTraversal, capitalAnalysis, functionAnalysis, owner resolution
         │
    ┌────┴────┐
    ▼         ▼
HTTP API     STAGE 2 — Review Compiler (reviewCompiler.ts)
/admins      + descriptions + funds + resources + audits + governance + activity
/dependencies → compiled-review.json
              │
              ▼
STAGE 3 — Index Aggregation (defiscan-frontend/scripts/compile-data.ts)
  Cross-protocol totals, dependency deduplication, global stats
  → index.json
              │
              ▼
STAGE 4 — Frontend (defiscan-frontend)
  Static rendering from compiled-review.json + index.json
```

### Stage 1 — ProjectAnalysis

`ProjectAnalysis` is the single backend authority for admin and dependency computation. All consumers — the researcher UI, the review compiler, and the public frontend (transitively) — obtain admin and dependency data from here.

- **`getAdmins(contractFilter?)`** — groups permissioned functions by resolved owner address. Resolves `ownerDefinitions` from `functions.json` against `discovered.json`, enriches with capital analysis (enhanced-graph forward BFS), computes ownership chains (backward BFS), and annotates each admin with contract-tag attributes (`isExternal`, `isGovernance`, `entity`).
- **`getDependencies(contractFilter?)`** — groups external dependencies by address with per-function capital data. Dependencies come from three sources: call-graph detection, write-dependencies (external owners of permissioned functions), and manual entries.

Both methods accept an optional `contractFilter` for per-contract queries. The backend still builds the full graph (sub-second) but filters the response, so the same code path serves both project-wide and per-contract consumers.

See [Scoring & Review: ProjectAnalysis](features/scoring-and-review.md#projectanalysis-api) for the full type definitions.

### Stage 2 — Review Compiler

`reviewCompiler.ts` is a **thin assembly layer**. It calls `ProjectAnalysis` internally, overlays human-written descriptions from `review-config.json`, plus resources, audits, governance, and the activity feed, and writes a self-contained `compiled-review.json`. If data needs to be computed differently, the change belongs in `ProjectAnalysis`, not in the compiler.

The compiled artifact carries a **three-timestamp model** rather than a single "last compiled" time: `publishedAt` (first publication of the review, preserved forever), `lastModified` (max of researcher edit times across `review-config.json` / `resources.json` / `governance.json`), and `compiledAt` (live on-chain data freshness, sourced from `discovered.json.timestamp` — **not** wall-clock compile time, so a researcher-triggered recompile without a fresh discovery keeps it frozen). A separate "latest on-chain activity" signal is derived on the frontend from the newest `activity[]` event via `getLatestActivityTimestamp(review)` — it is not one of the three compiled fields. See [Scoring & Review: Timestamps](features/scoring-and-review.md#timestamps-three-timestamp-model) for the full field-by-field breakdown and consumer sites.

### Stage 3 — Index Aggregation

`defiscan-frontend/scripts/compile-data.ts` reads every `compiled-review.json` and produces `index.json` with cross-protocol totals (capital, functions, admins, dependencies), aggregate fund values, and a global dependency map keyed by address.

### Stage 4 — Frontend

The frontend reads `compiled-review.json` (per protocol) and `index.json` (cross-protocol) as static JSON. No client-side joining — all computation is pre-baked by the compiler. See `defiscan-frontend/src/types.ts` for the types consumed by the frontend.

### Funds vs Token Value

Two distinct value types flow through the entire pipeline and are always tracked separately:

| Concept | Source | Display color |
|---|---|---|
| **Funds** (capital at risk) | `balances + positions + aggregate` | Green |
| **Token Value** (market cap) | `tokenInfo.tokenValue = supply × price` | Yellow |

Aggregate funds (from factory contracts such as Uniswap V2) are included in `totalCapitalAtRisk` at the index aggregation stage.
