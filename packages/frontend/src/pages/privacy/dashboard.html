Methodology

How each dimension is evaluated. Adapted from L2BEAT's risk rosette framework for rollups.

1. Privacy Guarantees

How strong is the cryptographic privacy provided?

GREEN Full cryptographic privacy via ZK proofs. Sender, receiver, and amounts hidden. Anonymity set grows over time. Variable amounts supported.
YELLOW Cryptographic privacy with meaningful limitations: fixed denominations, partial metadata exposure, operator visibility, or deposits public.
RED No cryptographic privacy. Privacy is encryption-based only (breakable with key compromise) or operational only.

2. Fund Safety

Can funds be stolen, locked, or drained by any party?

GREEN Core contracts immutable or non-upgradeable for fund logic. No admin can drain funds. No material past exploits.
YELLOW Core fund logic secure, but upgradeability via governance (DAO + timelock). Or: ZK proofs prevent theft but operator can freeze funds. Fallback exit exists.
RED Funds can be frozen/seized/lost through validator collusion, insolvency, admin action, or known unpatched vulnerabilities.

3. Censorship Resistance

Can deposits or withdrawals be blocked by any party?

GREEN No party can censor at protocol level. Permissionless deposit/withdrawal. Decentralized relayer/sequencer. Fallback to direct submission.
YELLOW Soft censorship possible by compliance layer, but guaranteed fallback exit exists (e.g., ragequit, forced inclusion).
RED Centralized parties can censor with no protocol-level fallback. Permissioned by design with no exit mechanism.

4. Trust Assumptions

What external trust is required?

GREEN Green trusted setup (L2BEAT framework). Contracts immutable. All data on-chain. No admin keys. Strong economic security or Ethereum-anchored.
YELLOW Adequate trusted setup. Upgradeable via governance with timelocks. Some centralized components with checks.
RED Significant trust in small validator set, centralized admin keys, reputation-only security, or history of exploits/insolvency.

5. Compliance Compatibility

Can users demonstrate regulatory compliance without sacrificing privacy?

GREEN Built-in compliance mechanism preserving privacy: viewing keys, ZK proofs of fund legitimacy, or programmable compliance.
YELLOW Partial compliance: voluntary public exit sacrificing privacy, or compliance mechanism exists but centralized.
RED No compliance mechanism. Protocol sanctioned or facing significant regulatory action.
